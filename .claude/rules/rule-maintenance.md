# ルール文書のメンテナンス原則

`.claude/rules/*.md` を追加・更新するときは以下に従う。

## 1 ファイル 1 トピック、40 行以内を目安

大きなルールは複数ファイルに分割する。読み手 (Claude セッション) は
description の 1 行と、必要なときだけ本文を読む。1 ファイルに複数の
無関係な原則が混在すると、不要な内容まで読まされて context が消費される。

## 「再利用可能な原則」を書く

ルールは将来の自分・他の Claude セッションが参照する **抽象化された原則**
であるべき。

```markdown
❌ アンチパターン (具体的すぎる):
2026-06-17 のセッションで pre-commit が落ちたため、main.py の `with open("r"`
を ruff が `with open(`に書き換えた

✅ 修正パターン (再利用可能):
ruff format が touch する範囲は `cg/` / `kaggle_data/` / `.venv/` 除外の
全 .py。format で書き換わったら副作用がないか確認してから取り込む
```

Issue 番号と日付は git log で追える。ルール本文には残さない。

## frontmatter

```yaml
---
description: いつこのルールが効くか、何を防ぐかを 1 行で
paths: "glob,pattern1,pattern2"  # 任意。このルールが特に効くファイルパターン
---
```

`description` はこのルールが「適用される場面」を表す。
作業中のファイルが `paths` に該当するときは body を必ず読む。

## How to apply (ルールを書くときの自己チェック)

- 不変条件 (Invariants) と、それが必要な理由 (なぜ) と、適用手順
  (How to apply) を分けて書く
- 「不変条件」は短い・行頭が `- ` の bullet list 形式が読みやすい
- 「なぜ」は **過去の事故か、エンジン側の制約** を 1 段落で書く。
  ふんわりとした best practice は書かない
- 「How to apply」はチェックリスト形式。コマンド例があれば添える
- 既存ルールを更新するときは git history で前回の改訂理由を確認してから
  上書きする (前の運用者が試行錯誤の末たどり着いた表現かもしれない)
- 新規ルールを増やす前に、既存ルールに追記で済まないか検討する。
  ルールファイル数を抑えると "どこに何が書いてあるか" の検索コストが下がる
